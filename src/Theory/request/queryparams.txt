

What are Query Parameters?
✅ Definition
Query parameters are key-value pairs sent in the URL after ? to pass optional data to the server.
/api/v1/users?filter=name&value=aarav
here :- filer=name and value=aarav
these are query params 
structure of url :-
http://localhost:3000/api/v1/users?sort=age&order=desc
/api/v1/users → route/path
?sort=age&order=desc → query string

3) Query Params vs Route Params (VERY IMPORTANT INTERVIEW QUESTION)
This is one of the most asked backend interview questions.
✅ Route Params (req.params)
Used when resource identity is part of the path.
Use case:
Fetch a specific resource
/users/:id
/products/:productId
✅ Query Params (req.query)
Used for optional modifiers of a request.
Use case:
Filtering
Sorting
Searching
Pagination
Interview one-liner answer:- 
Route params identify a specific resource, while query params modify how a collection/resource is fetched.

Important: Query Params Are Always Strings (VERY IMPORTANT)
Interview answer:
In Express, req.query values are strings by default, so numbers, booleans, and arrays often need parsing/validation before use.

Why use query params for filtering instead of body?
Best answer:
Because:
GET requests are meant for reading/fetching data
Query params are visible, cacheable, bookmarkable
REST convention says:
GET /users?role=admin&page=2


Interview answer:
For read operations, query params are preferred because they align with REST semantics, are cache-friendly, easy to share, and naturally represent optional filters.

Q1. What are query parameters?
Answer:
Query parameters are optional key-value pairs appended to the URL after ?, used to modify how the server processes a request, commonly for filtering, sorting, searching, and pagination.

Q2. What is the difference between query params and route params?
Answer:
Route params identify a specific resource and are part of the URL path, while query params are optional modifiers used to filter or customize the response.

Q3. Are query params strings in Express?
Answer:
Yes, values in req.query are strings by default, so they should be parsed or validated when numbers, booleans, or arrays are expected.

Q4. When would you return 400 for query params?
Answer:
Return 400 Bad Request when query parameters are malformed, invalid, or violate API rules—for example invalid sort fields, non-numeric page values, or unsupported filters.

Q5. If filter returns no records, should you return 404 or 200?
Answer:
For collection filtering/search, return 200 OK with an empty array because the request is valid but no records matched. 404 is more appropriate when requesting a specific resource like /users/:id.

Q6. Why use query params for pagination?
Answer:
Pagination is an optional way to shape the result set of a collection, so query params are a natural fit: page, limit, offset, etc.


Q7. Can query params be used with POST requests?
Answer:
Yes, technically query params can be used with any HTTP method, but they are most commonly used with GET for retrieval modifiers. For POST, request body usually carries the main payload.

This is where interview level goes up.
If using MongoDB/Mongoose:
Example:
/users?isActive=true&page=1&limit=10
const page = Number(req.query.page) || 1;
const limit = Number(req.query.limit) || 10;
const skip = (page - 1) * limit;
const filter = {};
if (req.query.isActive) {
  filter.isActive = req.query.isActive === "true";
}
const users = await User.find(filter).skip(skip).limit(limit);
🎯 Interview insight:

In production, query params are typically translated into database filters, sort clauses, and pagination controls rather than filtering in memory.

16) Great Interview Answer Example (Use This)
If interviewer asks:
“How do you usually design APIs using query parameters?”
Strong answer:
For collection endpoints, I use query parameters for optional retrieval controls like filtering, searching, sorting, and pagination. I validate allowed fields, parse numeric/boolean values explicitly since req.query is string-based in Express, and map those parameters into database-level filters instead of in-memory operations. For invalid query combinations or unsupported filters, I return 400 Bad Request. If no records match, I usually return 200 OK with an empty array.

17) Important Coding Pattern (Interview Ready)
app.get("/api/v1/users", (req, res) => {
  const {
    search,
    sortBy = "name",
    order = "asc",
    page = "1",
    limit = "10",
  } = req.query;

  const allowedSortFields = ["name", "age"];

  if (!allowedSortFields.includes(sortBy)) {
    return res.status(400).json({
      message: "Invalid sort field",
    });
  }

  const pageNum = Number(page);
  const limitNum = Number(limit);

  if (isNaN(pageNum) || isNaN(limitNum) || pageNum < 1 || limitNum < 1) {
    return res.status(400).json({
      message: "Invalid pagination values",
    });
  }

  let result = [...users];

  if (search) {
    result = result.filter((user) =>
      user.name.toLowerCase().includes(search.toLowerCase())
    );
  }

  result.sort((a, b) => {
    if (typeof a[sortBy] === "string") {
      return order === "asc"
        ? a[sortBy].localeCompare(b[sortBy])
        : b[sortBy].localeCompare(a[sortBy]);
    }

    return order === "asc" ? a[sortBy] - b[sortBy] : b[sortBy] - a[sortBy];
  });

  const start = (pageNum - 1) * limitNum;
  const paginated = result.slice(start, start + limitNum);

  return res.status(200).json({
    total: result.length,
    page: pageNum,
    limit: limitNum,
    data: paginated,
  });
});

21) 2–3 YOE Interview Answer (Memorize This)
If they ask:

“Explain query params in backend APIs”
Use this:
Query parameters are optional key-value pairs appended to the URL after ?, mainly used in collection endpoints for filtering, searching, sorting, and pagination. In Express, they are accessed via req.query and arrive as strings, so I usually validate and parse them before use. For production APIs, I map query params into database queries instead of filtering in memory. I also whitelist allowed fields for dynamic filters, return 400 for invalid query inputs, and return 200 with an empty array when a valid filter yields no results.
