1) What are Route Parameters (req.params)?
✅ Definition

Route parameters are dynamic values embedded inside the URL path and are used to identify a specific resource.
Example:
/api/v1/users/101
If route is: app.get("/api/v1/users/:id", ...)

Then:
req.params.id // "101"
🎯 Interview one-liner:
Route parameters are dynamic path segments used to identify specific resources in REST APIs, and in Express they are accessed via req.params.

2) Why do we use req.params?
Used when the client wants one exact resource or a resource related to another resource.

Common use cases
1. Get user by ID
/users/10

2. Delete product by ID
/products/55

3. Update order by ID
/orders/999

4. Nested resources
/users/10/posts/5

3) Syntax in Express
Route:
app.get("/api/v1/users/:id", (req, res) => {
  console.log(req.params);
});
Request:
/api/v1/users/101
Output:
{ id: "101" }

4) Important: req.params Values Are Also Strings (VERY IMPORTANT)
Just like req.query, route params come as strings.
Example:
/users/10
req.params.id // "10"
⚠️ Not number 10

Why this matters?
If you compare with strict equality:
user.id === req.params.id
and:
user.id is number 10
req.params.id is string "10"
Then:
10 === "10" // false
✅ Fix
const id = Number(req.params.id);
🎯 Interview answer:
In Express, route params are strings by default, so I explicitly parse them when numeric IDs are expected to avoid type mismatch bugs.


5) req.params vs req.query (Most Asked)

This is a must-know question.

req.params
Used for specific resource identity
/users/10
req.params.id

req.query
Used for optional modifiers
/users?page=2&limit=10
req.query.page
🎯 Strong interview answer:
req.params is used when the resource identity is part of the URL path, like /users/:id, while req.query is used for optional request modifiers like filtering, sorting, and pagination.

6) Real REST API Meaning (Important for 3 YOE)
Good REST design:
Single resource:
/users/10
Collection with modifiers:
/users?page=1&limit=10
❌ Weak design:
/users?id=10

Technically works, but not ideal for REST.
🎯 Best interview answer:
For fetching a specific resource, I prefer route params like /users/:id because it clearly models resource identity in RESTful APIs. Query params are better for optional collection-level operations.


7) Basic Example — Get User by ID
app.get("/api/v1/users/:id", (req, res) => {
  const id = Number(req.params.id);
  const user = users.find((eachUser) => eachUser.id === id);
  if (!user) {
    return res.status(404).json({
      message: "User not found",
    });
  }
  return res.status(200).json(user);
});
8) Important Interview Question:
If ID does not exist, what status code?
✅ Correct answer:
404 Not Found
Because client asked for one specific resource.

Example:
/users/999
If no such user:
return res.status(404).json({
  message: "User not found",
});
🎯 Interview answer:
For GET /users/:id, if the resource doesn’t exist, I return 404 Not Found because the request is valid but the specific resource is missing.

9) Common Interview Questions on req.params
Q1. What is req.params?
Answer:
req.params contains route parameters extracted from dynamic URL segments defined in Express routes, such as /users/:id.

Q2. Are route params strings in Express?
Answer:
Yes, route params are strings by default, so I parse them when I expect numeric IDs or other specific types.

Q3. When should you use req.params instead of req.query?
Answer:
I use req.params when identifying a specific resource or hierarchical resource path, and req.query when modifying how a collection is fetched.

Q4. What should happen if the route param ID is invalid?
Answer:
I validate the ID format first. If it’s malformed, I return 400 Bad Request; if it’s valid but no resource exists, I return 404 Not Found.

🔥 This is a very strong 3 YOE answer.
10) Very Important Distinction:
400 vs 404 with req.params
This is a favorite interviewer question.
Case 1: Invalid format → 400 Bad Request
Example:
/users/abc
If ID should be numeric:
const id = Number(req.params.id);
if (isNaN(id)) {
  return res.status(400).json({
    message: "Invalid user id",
  });
}

Case 2: Valid format but not found → 404 Not Found
Example:
/users/999
ID is valid number, but no user exists.
return res.status(404).json({
  message: "User not found",
});
🎯 Best interview answer:
I separate malformed identifiers from missing resources. If the path parameter format is invalid, I return 400. If the format is valid but the resource doesn’t exist, I return 404.
This answer sounds experienced 💯

11) Best Practice — Validate Route Params
Never trust req.params.
Example:
app.get("/api/v1/users/:id", (req, res) => {
  const id = Number(req.params.id);
  if (isNaN(id) || id <= 0) {
    return res.status(400).json({
      message: "Invalid user id",
    });
  }
  const user = users.find((u) => u.id === id);
  if (!user) {
    return res.status(404).json({
      message: "User not found",
    });
  }
  return res.status(200).json(user)
  ;
});



12) Multiple Route Params (Important)
Express supports multiple dynamic segments.
Example:
/users/10/posts/5
Route:
app.get("/users/:userId/posts/:postId", (req, res) => {
  console.log(req.params);
});
Output:
{
  userId: "10",
  postId: "5"
}
🎯 Interview answer:
Route params can represent hierarchical resources, for example /users/:userId/posts/:postId, which is useful in nested REST API design.

16) Common Mistakes with req.params

❌ Mistake 2: Not validating malformed IDs

❌ Mistake 3: Returning 404 for invalid ID format
Wrong in many cases.
malformed ID → 400

17) Strong Production-Level Example (Interview Ready)
app.get("/api/v1/users/:id", (req, res) => {
  const id = Number(req.params.id);
  // Validate param format
  if (isNaN(id) || id <= 0) {
    return res.status(400).json({
      success: false,
      message: "Invalid user id",
    });
  }
  // Find resource
  const user = users.find((eachUser) => eachUser.id === id);
  // Resource missing
  if (!user) {
    return res.status(404).json({
      success: false,
      message: "User not found",
    });
  }
  // Success
  return res.status(200).json({
    success: true,
    data: user,
  });
});



18) Route Params with MongoDB / Mongoose (3 YOE level)

If using MongoDB, route params are often IDs like:
/users/67f31ab9280b9f7b9c4d1234
Interview trap:

MongoDB ObjectId is not numeric.
So validation is different.
Example:
import mongoose from "mongoose";
if (!mongoose.Types.ObjectId.isValid(req.params.id)) {
  return res.status(400).json({ message: "Invalid user id" });
}
🎯 Strong answer:

If I’m using MongoDB, I validate req.params.id with mongoose.Types.ObjectId.isValid() before querying, so malformed ObjectIds return 400 instead of causing cast errors.

This is excellent interview material.

19) Security / Stability Angle (Advanced)
Interviewers may ask:
“Why validate route params if DB will fail anyway?”
Strong answer:
I validate route params early to avoid unnecessary database calls, prevent noisy cast errors, improve API consistency, and return predictable client-facing status codes like 400 for malformed identifiers.
🔥 Very strong.

20) req.params vs req.body vs req.query (Must Memorize)
Feature	req.params	req.query	req.body
Source	URL path	URL after ?	Request payload
Use	Specific resource ID	Filtering/sorting/pagination	Create/update data
Example	/users/:id	/users?page=1	{ name: "Aarav" }
Usually Required?	Often yes	Usually optional	Usually yes for POST/PATCH
Type in Express	String	String	Depends on parser
🎯 Best one-liner:
req.params identifies the target resource, req.query shapes retrieval, and req.body carries the payload.

22) Best 3 YOE Interview Answer (Memorize This)
If interviewer asks:

“Explain req.params and how you use it in backend APIs.”
Use this:
In Express, req.params contains dynamic path values from routes like /users/:id, and I mainly use it to identify specific resources in REST APIs. Since route params are strings by default, I parse or validate them depending on the ID type—for example numeric IDs or MongoDB ObjectIds. I usually return 400 Bad Request for malformed identifiers and 404 Not Found when the identifier is valid but the resource doesn’t exist. 23) Coding Pattern You Should Practice
app.patch("/api/v1/users/:id", (req, res) => {
  const id = Number(req.params.id);

  if (isNaN(id) || id <= 0) {
    return res.status(400).json({
      message: "Invalid user id",
    });
  }

  const user = users.find((u) => u.id === id);

  if (!user) {
    return res.status(404).json({
      message: "User not found",
    });
  }

  const { name, age } = req.body;

  if (name) user.name = name;
  if (age) user.age = age;

  return res.status(200).json({
    message: "User updated successfully",
    data: user,
  });
});
